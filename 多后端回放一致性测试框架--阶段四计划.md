# 多后端回放一致性测试框架--阶段四计划

更新时间：2026-07-01

## 阶段目标

阶段四在阶段三基础 replay matrix 上补齐 Go 版特有的 `Summary` 与 `Track` 风险面。

本阶段继续只在 `test/replay_consistency_test.go` 内扩展测试内部 harness，不新增运行时 public API，不接入 Redis/Postgres/MySQL/ClickHouse 等外部后端。runnable matrix 仍固定为：

- `in_memory`
- `sqlite`

阶段四的验收重点：

- replay case schema 能表达 summary 与 track。
- runner 能真实调用 `CreateSessionSummary`、`GetSessionSummaryText`、`AppendTrackEvent`。
- snapshot/diff 已有的 summary 与 tracks 比较能力被真实 case 覆盖。
- 公开 replay cases 总数从 6 条扩展到至少 10 条。
- 正常矩阵 report 仍为 `[]`。

## Schema 扩展

在 `replayCase` 中新增两个字段：

```go
summaries []summaryStep
tracks    []trackSpec
```

新增类型：

```go
type summaryStep struct {
	name      string
	filterKey string
	force     bool
	text      string
	wantText  string
}

type trackSpec struct {
	name      string
	payload   map[string]any
	timestamp time.Time
}
```

字段语义：

- `summaryStep.name`：只用于失败信息定位。
- `summaryStep.filterKey`：传给 `CreateSessionSummary` 与 `GetSessionSummaryText`；空字符串表示 `session.SummaryFilterKeyAllContents`。
- `summaryStep.force`：默认建议 case 中显式写 `true`，避免 ShouldSummarize/增量边界导致不更新。
- `summaryStep.text`：执行前写入当前 backend 的 `deterministicSummarizer.text`，确保两个后端生成同一文本。
- `summaryStep.wantText`：可选；为空时默认等于 `text`。
- `trackSpec.name`：转成 `session.Track`。
- `trackSpec.payload`：runner 内 marshal 成 `json.RawMessage`，snapshot 已做 canonical JSON。
- `trackSpec.timestamp`：必须使用固定时间，避免真实时间差异。

可选 helper：

```go
func replaySummary(filterKey, text string, opts ...func(*summaryStep)) summaryStep
func withReplaySummaryForce(force bool) func(*summaryStep)
func withReplaySummaryWantText(text string) func(*summaryStep)

func replayTrack(name string, index int, payload map[string]any) trackSpec
```

`replayTrack` 默认 timestamp 使用 `replaySpecTime(100 + index)`，确保和 event timestamp 分区清晰且稳定。

## Runner 接入计划

在 `runReplayCaseOnBackend` 中，保持阶段三既有顺序，并在生成最终 snapshot 前插入 track 与 summary 驱动：

1. `CreateSession`
2. `UpdateAppState`
3. `UpdateUserState`
4. `UpdateSessionState`
5. 逐条 `AppendEvent`
6. 执行 `tc.tracks`
7. 执行 memory ops
8. 执行 memory queries
9. 执行 `tc.summaries`
10. `GetSession`
11. `ReadMemories`
12. `makeReplaySnapshot`

### Track 执行细节

新增 helper：

```go
func appendReplayTrack(
	t *testing.T,
	ctx context.Context,
	backend backendBundle,
	key session.Key,
	spec trackSpec,
)
```

执行流程：

- 每个 track append 前重新 `GetSession(ctx, key)`，避免持有旧 session 对象。
- `payload` 用 `json.Marshal(spec.payload)` 编码。
- 调用：

```go
backend.trackService.AppendTrackEvent(ctx, got, &session.TrackEvent{
	Track:     session.Track(spec.name),
	Payload:   payload,
	Timestamp: spec.timestamp,
})
```

注意：

- track 顺序要固定，因为 track index 会写入 `sess.State["tracks"]`，snapshot 的 `state` 和 `tracks` 都会比较它。
- 同一 track 多事件要按 spec 顺序 append。
- 多 track case 要明确 append 顺序，避免 map 遍历参与执行路径。

### Summary 执行细节

新增 helper：

```go
func createReplaySummary(
	t *testing.T,
	ctx context.Context,
	backend backendBundle,
	key session.Key,
	spec summaryStep,
)
```

执行流程：

- 每个 summary step 前重新 `GetSession(ctx, key)`。
- 设置 `backend.summarizer.text = spec.text`。
- 调用：

```go
backend.sessionService.CreateSessionSummary(ctx, got, spec.filterKey, spec.force)
```

- 再次 `GetSession(ctx, key)`。
- 调用 `GetSessionSummaryText` 做 sanity check：
  - `filterKey == session.SummaryFilterKeyAllContents` 时不传 option，或显式传 `session.WithSummaryFilterKey("")`。
  - 非空 filter key 时传 `session.WithSummaryFilterKey(spec.filterKey)`。
- 断言返回 `ok == true`。
- 断言返回文本等于 `spec.wantText`；若 `wantText == ""`，则等于 `spec.text`。

注意：

- 直接调用 `CreateSessionSummary`，不走 `EnqueueSummaryJob`，避免 async worker 和 cascade 并发引入不确定性。
- `CreateSessionSummary` 对非空 filter key 不会自动 cascade full summary；这是 `EnqueueSummaryJob/CreateSessionSummaryWithCascade` 的职责。本阶段 case 按直接调用语义设计。
- `UpdatedAt` 仍由 snapshot 归一化为 `updated_at_non_zero`，不比较真实时间。
- boundary 中的 `LastEventID` 来自 replay event 固定 ID，理论上应跨后端一致；若不一致，优先检查 runner 或 normalize，而不是添加 allowed diff。

## 新增 Replay Cases

在 `basicReplayCases()` 继续追加至少 4 条 case，使公开 case 总数达到 10 条。

### 7. full_summary

目标：

- 覆盖 full-session summary。
- 验证 `Session.Summaries[""]`、summary text、boundary metadata 与 `GetSessionSummaryText`。

建议内容：

- events：
  - user: `"summarize this session"`
  - assistant: `"summary source answer"`
- summaries：
  - `filterKey: session.SummaryFilterKeyAllContents`
  - `force: true`
  - `text: "full summary for replay"`

期望：

- snapshot `summary[""].summary == "full summary for replay"`。
- boundary filter key 为空，version 非零，last event ID 稳定。

### 8. filter_key_summary

目标：

- 覆盖 branch/filter-key summary。
- 验证指定 filter key 的 summary 不依赖 full summary。

建议内容：

- events：
  - root 分支 user/assistant
  - `root/tools/weather` 分支 tool call/tool result
  - `root/tools/calendar` 分支另一个 assistant event
- summaries：
  - `filterKey: "root/tools/weather"`
  - `force: true`
  - `text: "weather branch summary"`

期望：

- snapshot 只要求 `summary["root/tools/weather"]` 存在。
- `GetSessionSummaryText(ctx, sess, session.WithSummaryFilterKey("root/tools/weather"))` 返回 `"weather branch summary"`。
- 不依赖 fallback 到 full summary。

### 9. summary_overwrite_boundary

目标：

- 覆盖同一 filter key 多次 force update。
- 验证最后一次 summary 覆盖前一次。
- 验证 boundary metadata 仍可跨后端一致比较。

建议内容：

- events：
  - 两轮 root 对话，保证有稳定 last event。
- summaries：
  - 第一次：`filterKey: ""`，`force: true`，`text: "first full summary"`
  - 第二次：`filterKey: ""`，`force: true`，`text: "updated full summary"`

期望：

- snapshot 中 `summary[""].summary == "updated full summary"`。
- report 不出现 summary diff。

### 10. track_events

目标：

- 覆盖同一 track 多事件、多 track、payload canonical JSON、timestamp 顺序。

建议内容：

- events：
  - user 发起工具任务
  - assistant 返回执行结果
- tracks：
  - `tool.latency` event 1：`{"tool":"weather","latency_ms":42,"status":"started"}`
  - `tool.latency` event 2：`{"tool":"weather","latency_ms":87,"status":"finished"}`
  - `task.status` event 1：`{"task":"child-a","state":"done","error":null}`

期望：

- snapshot `tracks` 按 track name 排序稳定。
- 每个 track 内 event 顺序保持 append 顺序。
- payload JSON key 顺序不影响比较。

## 风险与处理策略

| 风险 | 处理策略 |
| --- | --- |
| summary boundary 的 `LastEventID` 跨后端不一致 | 先检查 replay event ID 是否固定、summary 是否基于最新 `GetSession`，再检查 normalize；不使用 allowed diff 放行 |
| `CreateSessionSummary` 非空 filter key 不生成 full summary | 按直接调用语义设计 case；如需 cascade，应显式改用 `EnqueueSummaryJob`，但阶段四不建议引入 async |
| `GetSessionSummaryText` 对缺失 filter key fallback 到 full summary | filter-key case 不创建 full summary，避免误把 fallback 当成功 |
| track 写入修改 `State["tracks"]` 导致 state diff | 固定 track append 顺序，确保两个后端 state 中 track index 一致 |
| payload map marshal 顺序不稳定 | snapshot 已 canonical JSON；case 仍使用简单 map，复杂 payload 留给可选补充 case |
| smoke test 中 track 使用 `time.Now()` | 不影响 matrix；本阶段新增 track case 必须使用固定 timestamp |
