# Design

This example demonstrates an auditable Evaluation + PromptIter regression loop with no external model dependency. The main fake mode still exercises the real runtime path: a candidate `llmagent` runs behind `runner.NewRunner`, `evaluation.AgentEvaluator` scores train and validation evalsets, and the PromptIter engine applies accepted patches through the existing profile surface path. The fake model and fake PromptIter workers are deterministic substitutes only for model inference and optimization worker behavior.

The target surface is the `lookup_record` tool declaration. Baseline wording intentionally under-describes flight lookup capability, so train and validation cases expose tool trajectory and final-response failures. PromptIter acceptance decides whether each round's patch becomes the engine's `AcceptedProfile` based on train and validation score gain. The final gate is separate: it checks the final accepted profile against validation delta, new hard fails, critical-case regression, latency, and zero fake-mode cost. This split shows why a patch can be accepted by PromptIter but still be rejected for release.

Failure attribution is computed from the final candidate validation failures. It first compares actual and expected invocations for missing tools, wrong tool names, argument mismatches, and final-response mismatches. If invocation data is missing, it falls back to metric reasons and terminal trace output for route, format, knowledge, or generic metric failures. The report stores per-case reasons, tool names, terminal trace details, score deltas, patches, call counts, prompt hash, and gate reasons so the optimization artifact can be reviewed without rerunning the loop.

Trace smoke is intentionally narrower. Its evalset uses `evalMode: "trace"` with recorded actual invocations and execution traces. The evaluator replays those records and skips runner inference, so trace smoke verifies trace-mode compatibility, metric evaluation, and attribution plumbing, but it cannot prove that a prompt patch changes future candidate behavior. The report therefore marks optimization as skipped with an explicit reason.
