# 多后端回放一致性测试框架--阶段五执行摘要

更新时间：2026-07-01

## 阶段目标

阶段五验证 replay consistency harness 能发现人工制造的不一致，而不是只验证正常路径。

本阶段覆盖：

- snapshot mutation 异常注入
- SQLite/public API 异常注入
- `allowed_diff` 显式规则收紧与专项验证
- report entry 字段完整性校验

本阶段仍只覆盖 `InMemory + SQLite`，不新增运行时 public API，不接入外部后端。

## 已完成改动

继续只扩展：

- `test/replay_consistency_test.go`

新增 helper：

- `refreshReplayCaseResultSnapshot`
- `runReplayCaseWithBackendInjection`
- `requireReplayDiff`
- `requireReplayReportFields`
- `replayCaseByName`

收紧 `allowedDiffRule`：

- `Section` 必须非空且不能是 `*`
- `Path` 必须非空且不能是单独 `*`
- `BackendA` / `BackendB` 必须非空且不能是 `*`
- `Reason` 必须非空白
- backend pair 仍支持左右顺序互换
- path 仍支持局部 glob，例如 `$.memory[*].content`

同时将 `diffEntry` 的 `left/right/reason/context` JSON 字段改为稳定输出，便于 report schema 校验。

- `TestReplayConsistencyAllowedDiffRules_RequireExplicitMatch`

- no rule

- backend wildcard rejected

- path mismatch
- backend mismatch
- valid path glob
- valid reversed backend pair

## 已验证行为

阶段五验证了：

- 正常矩阵仍为零 diff
- 人工注入异常能被 100% 检出
- 注入 diff 默认全部 unallowed
- report entry 包含 `case/session_id/backend_a/backend_b/section/path/left/right/allowed/reason/context`
- summary filter-key 错误能定位 `summary_filter_key`
- track payload/order 错误能定位 `track_name` 与 `track_event_index`
- memory pollution 能携带 `memory_key` / raw ID context
- env override report 路径继续可用
- 仓库根目录未生成 `session_memory_summary_track_diff_report.json`

## 遇到的问题与处理

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| `partial_event_loss` 初版 mutation 把 `Events` 设为 `nil`，diff path 落在 `$.events` 且无 `event_index` | nil slice 和 non-empty slice 在 JSON diff 中变成整体值差异 | 改为 `[]replayEventSnapshot{}`，触发 list item missing diff，path 为 `$.events[0]` 并携带 `event_index=0` |
| report 字段校验要求 `reason/context` 稳定存在 | 旧结构体用了 `omitempty`，unallowed diff 没有 reason 时会省略字段 | 移除 `omitempty`，让字段稳定输出 |
