## Normalize 策略

Events：

- 通过 `json.Marshal(event.Event)` 保留 Go 当前事件 JSON 语义
- 删除生成型字段：
  - event 顶层 `id`
  - event 顶层 `timestamp`
  - event 顶层 `created`
  - nested `response.id`
  - nested `response.timestamp`
- 保留业务语义字段，例如：
  - `role/content/tool_calls/tool response`
  - `branch`
  - `filterKey`
  - `tag`
  - `stateDelta`
  - `extensions`
  - `actions`
- `StateDelta` 走 state normalize，避免 `[]byte` 导致误报

State：

- `[]byte` 优先按 JSON 解析
- JSON 解析成功后转为 canonical JSON 结构
- JSON 解析失败但 UTF-8 合法时按字符串比较
- 非 UTF-8 二进制按 base64 结构比较

Memory：

- 比较内容语义，不把后端生成的 memory ID 当等值字段
- 保留 raw ID 在 snapshot 内部字段中，用于 report context 定位
- topics、participants 排序
- event time 归一化为 UTC RFC3339Nano
- 按 canonical memory key 排序，减少后端返回顺序差异

Summary：

- 按 filter-key map 比较
- 比较 summary text、topics、boundary
- `UpdatedAt` 只比较是否非零：`updated_at_non_zero`
- boundary 比较：
  - version
  - filter_key
  - cutoff_at
  - last_event_id

Tracks：

- 按 track name 排序
- 同一 track 内按事件顺序比较
- payload 按 canonical JSON 比较
- timestamp 归一化为 UTC RFC3339Nano

## Diff / Report 能力

核心 helper：

- `makeReplaySnapshot(sess, memories)`
- `diffReplaySnapshots(caseName, sessionID, backendA, backendB, left, right, allowedRules)`
- `writeReplayDiffReport(path, entries)`
- `replayDiffReportPath()`

recursive diff：

- map key 排序后比较
- slice 按 index 比较
- 缺失字段用 `{"replay":"missing"}` 表示
- path 使用 JSONPath-like 格式，例如：
  - `$.events[0].author`
  - `$.state.json.a`
  - `$.memory[0].content`
  - `$.summary["branch/a"].summary`
  - `$.tracks[0].events[0].payload.a`

report path：

- 默认：`../session_memory_summary_track_diff_report.json`
- 从 `test` module 执行时，实际落到仓库根目录：
  - `D:\project\OpensourceTencent\trpc-agent-go\session_memory_summary_track_diff_report.json`
- 环境变量覆盖：
  - `TRPC_AGENT_REPLAY_REPORT_PATH`

allowed diff：

- 默认不允许任何 diff
- 只有显式 `allowedDiffRule` 命中时才设置：
  - `allowed=true`
  - `reason=<rule reason>`
- rule 支持：
  - `section`
  - `path`，支持 `*` 通配
  - `backend_a`
  - `backend_b`
  - `reason`
- backend pair 支持左右互换匹配

## 阶段三入口建议

阶段三应继续在 `test/replay_consistency_test.go` 扩展，不拆新包。

建议直接复用阶段二 helper：

1. 定义 `replayCase`
2. 定义 `eventSpec`
3. 定义 `memoryOpSpec`
4. 定义 `memoryQuerySpec`
5. 实现 case runner：
   - 对每个 backend 创建 session
   - append events
   - update session/app/user state
   - 执行 memory add/update/delete/search
   - 读取 session 和 memories
   - 调用 `makeReplaySnapshot`
6. 对 backend pair 调用 `diffReplaySnapshots`
7. 聚合所有 diff 后调用 `writeReplayDiffReport`
8. 无未允许 diff 时，report 内容应为 `[]`
9. 有未允许 diff 时，测试失败并打印 report path

阶段三第一批 case 建议只覆盖 events/state/memory：

- single turn
- multi turn
- tool call + tool response + args extension
- session/app/user/scoped state
- memory add/update/search
- interleaved child invocation / branch order

Summary 和 Track 的真实 replay case 留给阶段四更合适；阶段二当前比较器已经具备 summary/track 的 diff 能力。
