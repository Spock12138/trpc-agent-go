# 两主机工作区同步摘要：多租户与节点部署

## 目标

当前任务是实现“多租户与节点部署”题目。

两台主机轮换工作时，统一使用 GitHub fork 上的 workspace 分支作为同步中介：

```powershell
feat/multi-tenant-node-deployment-workspace
```

这个分支只作为“移动工作区”，用于同步代码、草稿、学习文档、阶段总结、实验记录和临时笔记。它不直接提交 PR。

最终提交上游时，从最新 `main` 新建干净 PR 分支：

```powershell
feat/multi-tenant-node-deployment
```

只把确认要进入上游的代码、测试和正式文档迁移到干净 PR 分支。个人笔记、草稿、阶段记录不进入最终 PR。

如果另一台主机上已经存在这个 worktree 或本地分支，不要重复创建，直接切换并拉取即可。

## 每次开始工作前

```powershell
cd D:\project\OpensourceTencent\trpc-agent-go-multi-tenant-workspace

git switch feat/multi-tenant-node-deployment-workspace
git fetch origin
git pull --ff-only
git status
```

确认看到类似状态后再开始开发：

```text
On branch feat/multi-tenant-node-deployment-workspace
Your branch is up to date with 'origin/feat/multi-tenant-node-deployment-workspace'.
nothing to commit, working tree clean
```

如果 `git pull --ff-only` 失败，通常说明两台主机都产生了提交。先不要强推，先查看状态：

```powershell
git status
git log --oneline --decorate --graph -20
```

没有把握时先停下，不要用 `git push --force`。

## 工作过程中

推荐小步提交，并尽量把代码和笔记分开。

代码变更：

```powershell
git add -p
git commit -m "feat: xxx"
```

测试或修复：

```powershell
git add -p
git commit -m "fix: xxx"
```

学习文档、阶段总结、计划记录：

```powershell
git add "*.md"
git commit -m "docs: update multi-tenant deployment notes"
```

临时保存：

```powershell
git add -A
git commit -m "wip: save multi-tenant deployment workspace progress"
```

如果某个 commit 同时包含代码和笔记，也可以先提交到 workspace 分支；最终整理 PR 时再拆分。

## 每次结束工作前

确认当前改动：

```powershell

```

如果还有未提交内容，先提交：

```powershell
git add -A
git commit -m "wip: save multi-tenant deployment workspace progress"
```

推送到远端，让另一台主机可以接手：

```powershell
git push
```

结束前再次确认：

```powershell
git status
```

理想状态是：

```text
On branch feat/multi-tenant-node-deployment-workspace
Your branch is up to date with 'origin/feat/multi-tenant-node-deployment-workspace'.
nothing to commit, working tree clean
```

如果 `git status` 显示没有改动，可以跳过 `add/commit`，只执行：

```powershell
git push
git status
```

## 前情资料同步建议

当前题目的前情资料可以提交到 workspace 分支，方便两台主机同步阅读，例如：

```text
开源实战--技术路线.md
开源实战题目.md
开源实战题目--讲解.md
开源实战选题上下文摘要.md
```

这些资料可以进入 workspace 分支，但默认不进入最终 PR 分支。

如果 `library/` 是学习资料且体积可控，可以提交到 workspace 分支；如果包含大文件、下载包、生成物或临时缓存，先不要直接加入 Git。

## 最终整理 PR 分支

不要从 `feat/multi-tenant-node-deployment-workspace` 直接提 PR。

等 workspace 实现稳定后，从最新官方主线新建干净 PR worktree：

```powershell
cd D:\project\OpensourceTencent\trpc-agent-go

git fetch origin --prune
git fetch upstream --prune

git worktree add ..\trpc-agent-go-multi-tenant-pr `
  -b feat/multi-tenant-node-deployment `
  upstream/main
```

从 workspace 分支挑选正式提交：

```powershell
cd D:\project\OpensourceTencent\trpc-agent-go-multi-tenant-pr

git log --oneline upstream/main..origin/feat/multi-tenant-node-deployment-workspace
```

如果某些提交只包含正式代码、测试或正式文档，可以直接 cherry-pick：

```powershell
git cherry-pick <commit-hash>
```

如果某个提交混有代码和笔记，使用不自动提交的方式：

```powershell
git cherry-pick -n <commit-hash>
```

然后只暂存要进入 PR 的文件：

```powershell
git restore --staged .
git status
git add -p <需要进入PR的文件或目录>
git commit -m "feat: add multi-tenant node deployment"
```

提交前检查不要混入个人笔记：

```powershell
git diff --name-only upstream/main...HEAD
git status
```

确认只包含应该提交给上游的代码、测试和正式文档后再推送：

```powershell
git push -u origin feat/multi-tenant-node-deployment
```

## 铁律

1. 开工前：`git pull --ff-only`
2. 收工前：`commit + push`
3. 不要从 `feat/multi-tenant-node-deployment-workspace` 直接提 PR
4. 最终 PR 从最新 `upstream/main` 新建干净分支 `feat/multi-tenant-node-deployment`
5. workspace 分支可以包含学习文档和草稿，PR 分支必须只包含正式代码、测试和正式文档
