# 多后端回放一致性测试框架--阶段一执行摘要

更新时间：2026-07-01

## 阶段目标

阶段一只完成最小可运行骨架：在 `test` module 中构造、使用并关闭 InMemory 与 SQLite 的 Session / Memory / Track 后端。不实现 snapshot、diff、report，也不添加正式 replay cases。

## 已完成改动

- 新增 `test/replay_consistency_test.go`。
- 更新 `test/go.mod`，加入 SQLite session/memory 子 module、`go-sqlite3` 以及 local `replace`。
- 运行 `go mod tidy` 后更新 `test/go.sum`。
- 未生成 `session_memory_summary_track_diff_report.json`。

## 新增测试骨架

`test/replay_consistency_test.go` 当前包含：

- `backendBundle`
- `deterministicSummarizer`
- `openSQLiteDB`
- `makeReplayBackends`
- `closeReplayBackends`
- `TestReplayConsistencySmoke_BackendsConstructUseAndClose`

`backendBundle` 字段：

- `name string`
- `sessionService session.Service`
- `trackService session.TrackService`
- `memoryService memory.Service`
- `summarizer *deterministicSummarizer`

`deterministicSummarizer` 已实现 `session/summary.SessionSummarizer`，默认 summary 文本为 `smoke summary`。

## 后端工厂现状

`makeReplayBackends(t)` 当前创建两个 backend：

- `in_memory`
  - `session/inmemory.NewSessionService`
  - `memory/inmemory.NewMemoryService`
- `sqlite`
  - `session/sqlite.NewService`
  - `memory/sqlite.NewService`
  - session DB 和 memory DB 使用两个独立临时 SQLite 文件，避免两个 service 共同 owning 同一个 `*sql.DB`

所有 backend 通过 `t.Cleanup` 注册 `closeReplayBackends`，先关 memory service，再关 session service。

## Smoke Test 覆盖内容

`TestReplayConsistencySmoke_BackendsConstructUseAndClose` 对每个 backend 执行：

- `CreateSession`
- `GetSession`
- 校验 session state `stage=smoke`
- `AddMemory`
- `SearchMemories`
- `AppendTrackEvent`
- 再次 `GetSession` 并校验 `smoke-track` 存在

这个测试只验证基础后端可用，不做跨后端等值比较。
