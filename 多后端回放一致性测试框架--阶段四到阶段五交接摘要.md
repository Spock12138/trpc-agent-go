## 第四步踩坑

### Summary 时间边界

初版 replay event timestamp 固定在 `2026-07-01 02:00 UTC`，可能早于测试运行时创建的 session `CreatedAt`。`GetSessionSummaryText` 会过滤 `UpdatedAt < CreatedAt` 的 summary，因此刚创建的 summary 会被判定为不存在。

最终处理：

- 使用 `time.Now().UTC().Add(24 * time.Hour).Truncate(time.Second)` 作为单次测试运行稳定基准。
- `replaySpecTime(index)` 在该基准上加秒偏移。

第五步注意：

- 不要把 replay 时间改回绝对历史时间。
- 注入 summary 异常时，如果 `GetSessionSummaryText` 失败，先检查 `UpdatedAt/CutoffAt` 与 `CreatedAt`，不要急着改 allowed_diff。

### Track 会污染 state

`AppendTrackEvent` 会维护 session state 内的 track index，也就是 `state["tracks"]`。

第五步注意：

- track order/payload 异常可能同时出现在 `tracks` section 和 `state` section。
- 如果第五步只想验证 track payload drift，建议只 mutate snapshot 的 `Tracks[*].Events[*].Payload`，避免无关 state diff 干扰。

### 直接 Summary 不 Cascade

阶段四调用的是 `CreateSessionSummary`，不是 `EnqueueSummaryJob`。

结论：

- 非空 filter key summary 不会自动生成 full summary。
- `filter_key_summary` case 故意不依赖 full summary fallback。

第五步注意：

- wrong filter-key 注入要检查 `summary_filter_key` context。
- 不要假设 filter-key summary 一定伴随 `summary[""]`。

## 第五步建议顺序

推荐先做 snapshot mutation，后做真实 SQLite/public API 注入。

1. `summary_loss`
   - 从一侧 snapshot 删除 `Summary[""]` 或 `Summary["root/tools/weather"]`
   - 验证 section 为 `summary`
   - 验证 context 包含 `summary_filter_key`

2. `wrong_summary_filter_key`
   - 把一侧 `Summary["root/tools/weather"]` 移到错误 key
   - 验证缺失/新增 diff 都能定位 filter key

3. `track_payload_drift`
   - 修改一侧 `Tracks[*].Events[*].Payload`
   - 验证 section 为 `tracks`
   - 验证 context 包含 `track_name` 和 `track_event_index`

4. `track_order_drift`
   - 交换同一 track 下两个 event 的顺序
   - 验证 diff path 精确到 event index

5. `allowed_diff`
   - 显式命中 section + path glob + backend pair + reason 才 allowed
   - 默认业务字段差异必须 unallowed

6. 真实 SQLite/public API 注入
   - duplicate event
   - state pollution
   - memory pollution
   - summary overwrite

## 第五步审查清单

- 正常 case 仍保持 `allDiffs == nil/empty`
- 注入 case 必须先证明有 diff，再证明 context 精准
- report entry 必须包含：
  - `case`
  - `session_id`
  - `backend_a`
  - `backend_b`
  - `section`
  - `path`
  - `left`
  - `right`
  - `allowed`
  - `reason`
  - `context`
- summary/track 业务差异不要用 allowed_diff 放行
- ID/时间类差异优先修 normalize 或 runner
- 测试中继续使用 `TRPC_AGENT_REPLAY_REPORT_PATH` 指向临时目录
- 完成后确认仓库根目录没有默认 report 残留

## 第五步不应改动

- 不拆新 package
- 不新增运行时 public API
- 不接入外部后端
- 不把 `EnqueueSummaryJob` 引入正常 replay matrix
- 不把 `time.Now()` 直接写入 replay case 的 event/track timestamp
- 不用 allowed_diff 掩盖正常矩阵中的真实差异
