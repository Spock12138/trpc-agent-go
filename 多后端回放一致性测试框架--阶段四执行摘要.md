# 多后端回放一致性测试框架--阶段四执行摘要

更新时间：2026-07-01

## 阶段目标

阶段四在阶段三基础 replay matrix 上补齐 Go 特有的 `Summary` 与 `Track` 风险面。

本阶段覆盖：

- full summary
- filter-key summary
- summary overwrite/update 与 boundary metadata
- track event payload、timestamp、track name 与同 track 事件顺序

本阶段仍只覆盖 `InMemory + SQLite`，不新增运行时 public API，不接入外部后端。

## 已完成改动

继续只扩展：

- `test/replay_consistency_test.go`

新增/扩展测试内部 schema：

- `replayCase.summaries []summaryStep`
- `replayCase.tracks []trackSpec`
- `summaryStep`

新增 helper：

- `appendReplayTrack`
- `createReplaySummary`
- `replaySummary`
- `withReplaySummaryName`
- `withReplaySummaryForce`
- `withReplaySummaryWantText`
- `replayTrack`

新增稳定时间基准：

- `replayBaseTime = time.Now().UTC().Add(24 * time.Hour).Truncate(time.Second)`
- `replaySpecTime(index)` 改为基于 `replayBaseTime` 加秒偏移

这样 replay event/track timestamp 在单次测试运行内跨后端一致，同时保证 summary cutoff/UpdatedAt 不早于 session `CreatedAt`，避免 `GetSessionSummaryText` 的时间过滤误判。

## Runner 行为变更

`runReplayCaseOnBackend` 当前顺序：

1. `CreateSession`
2. `UpdateAppState`
3. `UpdateUserState`
4. `UpdateSessionState`
5. 逐条 `AppendEvent`
6. 逐条 `AppendTrackEvent`
7. 执行 memory ops
8. 执行 memory queries
9. 逐条 `CreateSessionSummary`
10. `GetSession`
11. `ReadMemories`
12. `makeReplaySnapshot`

Track 执行细节：

- 每个 track spec 执行前重新 `GetSession`
- `payload map[string]any` marshal 为 `json.RawMessage`
- 调用 `backend.trackService.AppendTrackEvent`
- timestamp 使用固定 spec 时间

Summary 执行细节：

- 每个 summary step 执行前重新 `GetSession`
- 设置当前 backend 的 `deterministicSummarizer.text`
- 调用 `backend.sessionService.CreateSessionSummary(ctx, got, filterKey, force)`
- 再次 `GetSession`
- 调用 `GetSessionSummaryText` 做 sanity check
- full summary 不传 `SummaryOption`
- filter-key summary 使用 `session.WithSummaryFilterKey(filterKey)`

## 新增 4 个公开 Case

### 7. full_summary

覆盖：

- full-session summary
- `Session.Summaries[""]`
- `GetSessionSummaryText`
- summary boundary version/filterKey/cutoff/last event ID

### 8. filter_key_summary

覆盖：

- `root/tools/weather` filter-key summary
- root/weather/calendar 多分支事件混合
- `GetSessionSummaryText(ctx, sess, session.WithSummaryFilterKey("root/tools/weather"))`
- 不依赖 full summary fallback

### 9. summary_overwrite_boundary

覆盖：

- 同一 full summary filter key 连续 force 两次
- 后一次 summary text 覆盖前一次
- boundary metadata 在 overwrite 后仍跨后端一致

### 10. track_events

覆盖：

- 同一 track 多事件：`tool.latency`
- 多 track：`tool.latency`、`task.status`
- payload canonical JSON
- fixed timestamp
- track event append order

公开 replay cases 总数已从 6 条增加到 10 条。

## 遇到的问题与处理

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| `GetSessionSummaryText` 找不到刚创建的 summary | 初版 replay timestamp 固定在 `2026-07-01 02:00 UTC`，可能早于运行时 session `CreatedAt`，summary text 读取逻辑会过滤 `UpdatedAt < CreatedAt` 的 summary | 改为基于 `time.Now().UTC().Add(24h).Truncate(time.Second)` 的单次运行稳定时间基准 |
| 辅助命令中 `gofmt` 路径错误 | 在 `test` 工作目录下传入了 `test\replay_consistency_test.go` | 后续从仓库根目录使用明确 Go 安装路径重新执行 `gofmt` |
| PowerShell 中一次 `rg` 模式被拆成路径 | 辅助查询命令的转义问题 | 未影响实现与测试，已忽略该辅助查询失败 |

## 阶段五入口建议

阶段五可基于当前真实 summary/track case 增加异常注入，不需要再改 runner 主流程。

建议优先做：

- snapshot mutation：
  - summary loss
  - wrong summary filter-key
  - track payload drift
  - track order drift
- report context 验证：
  - `summary_filter_key`
  - `track_name`
  - `track_event_index`
- allowed_diff 专项：
  - section/path/backend pair/reason 全部命中才 allowed
  - 默认不允许 summary/track 业务字段差异

注意：

- summary 差异仍应优先通过 runner/normalize 修正，不用 allowed diff 放行业务差异。
- track 异常注入需要同时关注 `tracks` section 与 `state["tracks"]` 的差异定位。
