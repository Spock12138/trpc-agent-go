## Runner 行为

`runReplayCaseOnBackend` 对每个 backend 执行：

1. `CreateSession`
2. `UpdateAppState`
3. `UpdateUserState`
4. `UpdateSessionState`
5. 逐条 `AppendEvent`
6. 执行 memory ops：
   - `AddMemory`
   - `UpdateMemory`
   - `DeleteMemory`
7. 执行 memory queries：
   - `SearchMemories`
8. `GetSession`
9. `ReadMemories`
10. `makeReplaySnapshot`

event append 前会重新 `GetSession`，避免持有旧 session 对象导致后端行为不一致。

memory update/delete 使用 alias 机制：

- add 后通过 `ReadMemories` 找到 content 对应 raw ID
- 保存到 `memoryAliases`
- update/delete 通过 alias 取 ID
- update 可用 `memory.WithUpdateResult` 更新 alias

## Matrix 比较

`TestReplayConsistencyMatrix_BasicCases` 当前流程：

1. 创建 `InMemory + SQLite` backends
2. 加载 `basicReplayCases`
3. 每个 case 在每个 backend 上运行
4. 对 backend pair 调用 `diffReplaySnapshots`
5. 聚合 diff
6. 存在未 allowed diff 时测试失败
7. 调用 `writeReplayDiffReport("", allDiffs)`
8. 断言 report JSON 为 `[]`
9. 断言 `allDiffs` 为空

注意：

- 测试中通过 `t.Setenv("TRPC_AGENT_REPLAY_REPORT_PATH", reportPath)` 把 report 写到临时目录。
- 这样仍覆盖正式 report path/env override 逻辑，但不会在仓库根目录留下 `session_memory_summary_track_diff_report.json`。

## 阶段四入口建议

阶段四目标是补 Go 特有的 Summary 与 Track 风险面。建议继续在同一文件扩展，不拆新包。

### 建议新增类型

在 `replayCase` 上追加：

- `summaries []summaryStep`
- `tracks []trackSpec`

建议定义：

```go
type summaryStep struct {
	filterKey string
	force     bool
	text      string
}

type trackSpec struct {
	name      string
	payload   json.RawMessage
	timestamp time.Time
}
```

也可以先更保守：

- summary text 继续用每个 backend 的 `deterministicSummarizer.text`
- track payload 用 `map[string]any`，在 runner 内 marshal

### Summary 接入位置

建议在 `runReplayCaseOnBackend` 中：

1. events/state/memory 全部执行后，重新 `GetSession`
2. 对每个 `summaryStep`：
   - 设置 `backend.summarizer.text`
   - 调用 `backend.sessionService.CreateSessionSummary(ctx, got, filterKey, force)`
   - 再次 `GetSession`
   - 可选调用 `GetSessionSummaryText` 做轻量 sanity check
3. 最后生成 snapshot

注意：

- 当前 fake summarizer 是 per-backend 实例，可安全设置文本。
- summary filter-key 应覆盖：
  - full summary：`session.SummaryFilterKeyAllContents`
  - branch summary：例如 `root/tools/weather`
  - overwrite/update：同一 filter key 调两次
- 阶段二 snapshot 已经比较：
  - `Session.Summaries[filterKey]`
  - summary text
  - topics
  - boundary version/filterKey/cutoff/lastEventID
  - updated_at_non_zero

### Track 接入位置

建议在 `runReplayCaseOnBackend` 中：

1. summary 前后都可；建议 events 后、summary 前写 track
2. 先 `GetSession`
3. 对每个 `trackSpec` 调用：
   - `backend.trackService.AppendTrackEvent(ctx, got, &session.TrackEvent{...})`
4. 最后重新 `GetSession`
5. 生成 snapshot

注意：

- track timestamp 使用 spec 固定时间，避免真实时间差异。
- payload 使用 canonical JSON。
- 阶段二 snapshot 已经比较：
  - track name
  - track event order
  - payload
  - timestamp

### 建议阶段四新增 case

建议至少新增 4 条，使公开 case 总数达到 10：

1. `full_summary`
   - full-session summary
   - `GetSessionSummaryText` sanity check

2. `filter_key_summary`
   - branch/filterKey summary
   - 确认不同 branch 不互相覆盖

3. `summary_overwrite_boundary`
   - 同一 filterKey force summary 两次

4. `track_events`
   - 同一 track 多个 event
   - 多 track
   - payload 包含工具耗时、状态、错误字段

如果想更贴近 plan，可再补：

5. `track_order_payload`
   - 验证 track order/payload drift 能被定位
